# Wine Gallery — обзор проекта (RU)

## 1) Что это за продукт
Wine Gallery — это веб‑приложение для интерактивной винной (и в будущем продуктовой) карты, заточенное под рестораны. Продукт показывает меню, поддерживает разные рестораны, языки, секции и deep‑links, и рассчитан на шаринг и работу менеджеров/сомелье.

## 2) Цели (стратегические)
- Масштабировать продукт на множество ресторанов.
- Дать управляемую, предсказуемую структуру меню.
- Обеспечить удобный UX для гостей и персонала.
- Подготовить архитектуру к переходу на роли и серверную авторизацию.

## 3) Ключевые задачи (ближайшие этапы)
- Ввести стартовые страницы ресторанов с логином/паролем.
- Поддержать несколько меню на ресторан (wine, desserts и т.д.).
- Реализовать брендинг и тему на уровне ресторана.
- Ввести красивую URL‑структуру вида /<restaurant>/<menu>.

## 4) Что уже есть (текущее состояние)
- Одностраничное приложение с UI карточек и секций.
- Конфиги на ресторан (configs/*.json).
- Работа с CSV как source of truth.
- Deep‑links через URL параметры (?section, ?w).
- Управление порядком секций через config (sectionsOrder, sectionsOrderByLang).

## 5) Основные правила (из SPEC.md)
- Не переименовывать файлы/папки без запроса.
- Не менять структуру проекта без разрешения.
- CSV — единственный источник данных.
- Секции управляются конфигом (sectionsOrder*).

## 6) Технические решения (текущие/планируемые)
### 6.1 Конфиги ресторанов
- Расширить configs/*.json следующими блоками:
  - auth (логины/роли)
  - branding (лого, названия)
  - theme (CSS‑переменные, шрифты)
  - menus (список меню и их CSV)

### 6.2 Авторизация (этапы)
- MVP: простая проверка логина/пароля на клиенте (низкая безопасность).
- Далее: серверная авторизация, роли, токены и логи.

### 6.3 Меню
- Одно меню = один CSV.
- Унифицированные колонки между ресторанами и меню.
- Единая навигация между меню.

### 6.4 URL
- Перейти на красивую структуру вида /restaurant/menu.
- Поддержать сохранение состояния (секция, модалка).

## 7) Риски и ограничения
- Клиентская авторизация не защищает от извлечения пароля.
- Без сервера невозможны полноценные роли и аудит.
- Рост количества ресторанов потребует более строгой схемы конфигов и валидации данных.

## 8) Куда идём дальше (предложенный порядок)
1) Спроектировать структуру расширенного конфига (auth/branding/theme/menus).
2) Ввести стартовую страницу ресторана с логином.
3) Реализовать множественные меню внутри ресторана.
4) Ввести красивую URL‑структуру.
5) Подготовить миграцию на серверную авторизацию.
